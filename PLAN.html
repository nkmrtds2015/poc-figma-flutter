Figma × Claude Code × Flutter 連携 PoC 計画

作成日: 2026-05-17  |  対象環境: Windows 11 + WSL2 (Ubuntu 24.04)

Context

目的: Figma でデザイン → Claude Code でコード生成 → Flutter でアプリ実行、という開発ワークフローを実証する PoC を作る。 最終的な本番ターゲットは組み込み Linux (詳細未確定) だが、今回はその前段として連携フローそのものを検証する。

  • Flutter 経験なし・Figma 未導入の状態からのゼロスタート
  • 開発環境: Windows 11 + WSL2 (Ubuntu 24.04)
    • WSL バージョン: 2.6.3.0
    • WSLg バージョン: 1.0.71 (GUI アプリ表示が追加設定なしで使える)
    • Windows バージョン: 10.0.26200 (Windows 11)
  • Claude Code は WSL2 上の Ubuntu で動作
  • Figma・ブラウザは Windows 側で操作
  • Claude Code が「設計・コード生成・レビュー」すべてを担う

0-2. Flutter SDK のインストール (WSL2 内)

snap は WSL2 では動作しないため、公式バイナリを手動展開する:

⚠️ 実績値: 実際にインストールした Flutter は 3.41.9（プランの 3.22.0 より新しい安定版）。 最新版バージョンは以下で確認してからダウンロードすること。 
# 最新安定版バージョンを確認
python3 -c "import json,urllib.request; d=json.load(urllib.request.urlopen('https://storage.googleapis.com/flutter_infra_release/releases/releases_linux.json')); r=[x for x in d['releases'] if x['channel']=='stable'][0]; print(r['version'], r['archive'])"

# 確認したバージョンでダウンロード（例: 3.41.9）
cd ~

source ~/.bashrc

flutter doctor

Windows 側の Flutter は使わない。組み込み Linux をターゲットにするため WSL2 内の Linux Flutter を使う。

0-3. Figma アカウント作成 & Figma MCP 設定

⚠️ 実績に基づく修正: Claude Code の MCP 設定は ~/.claude/settings.json ではなく ~/.claude.json (WSL 側) に保存される。 また VSCode Remote (WSL) 拡張経由で使う場合は ~/.vscode-server/ 内の Claude バイナリで登録する必要がある。
  1. Windows のブラウザで figma.com にアクセスし無料アカウント作成
  2. Figma Personal Access Token を発行 (Settings → Account → Personal access tokens)
    • Scope: File content: Read only のみ選択（Variables は無料プランでは表示されない）
    • Expiration: 30 days 推奨
  3. トークンをファイルに安全保存: echo 'FIGMA_ACCESS_TOKEN=<token>' > ~/.figma_token && chmod 600 ~/.figma_token
  4. MCP を登録（VSCode Remote WSL 環境の場合は vscode-server のバイナリを使う）
# 通常の WSL ターミナルから登録する場合
TOKEN=$(grep FIGMA_ACCESS_TOKEN ~/.figma_token | cut -d= -f2-)
claude mcp add figma -s user -- npx -y figma-developer-mcp --figma-api-key $TOKEN --stdio

# VSCode Remote (WSL) 拡張経由で使う場合は vscode-server のバイナリで登録
CLAUDE_BIN=$(ls ~/.vscode-server/extensions/anthropic.claude-code-*/resources/native-binary/claude 2>/dev/null | tail -1)
$CLAUDE_BIN mcp add figma -s user -- npx -y figma-developer-mcp --figma-api-key $TOKEN --stdio

# 登録確認
$CLAUDE_BIN mcp list   # figma: ✓ Connected が表示されれば OK

登録後、VSCode を完全再起動（Reload Window ではなく終了→再起動）すること。

Phase 3: Design Tokens 双方向同期 (1日)

⚠️ 方針変更 (2026-05-17): Figma の file_variables API スコープは エンタープライズプラン限定 のため、 無料プランでは API 経由の Variables 書き込みが不可。 Flutter → Figma の逆方向は Token Studio プラグインの手動 import で代替する。

Figma → Flutter（トークン順方向 / 自動）:

  1. Figma で Token Studio プラグインを開き、Variables を tokens.json として Export
  2. figma2flutter で Dart コードに変換
dart pub global activate figma2flutter
figma2flutter --input tokens.json --output lib/theme/tokens.dart

Flutter → Figma（トークン逆方向 / 手動操作）:

  1. Flutter 側で色などのトークン値を tokens.json に書き出すスクリプトを実行
  2. Windows 側の Figma で Token Studio プラグインを開き、tokens.json を手動 Import
  3. Figma Variables が更新されたことを確認
# tokens.json を手動で編集 or スクリプト生成後、
# Figma Desktop → Token Studio プラグイン → Import JSON → tokens.json を選択

完全自動化は有料プラン (Enterprise) が必要。無料プランでは手動 import ステップが残る。

Phase 6: Flutter → Figma 完全逆反映（保留中）

背景: 現在の Flutter → Figma 逆方向は色のみ自動・テキスト/サイズは手動。 figma-use (Chrome DevTools Protocol 経由) を使えば、テキスト・サイズ・色すべてを自動反映できる可能性がある。

提案フロー:

  1. Flutter 差分の把握 — git diff でテキスト・色・サイズの変更を抽出 → diff_result.json
  2. Figma データのエクスポート — Figma MCP / REST API で現在のノード情報を取得
  3. 差分反映スクリプト — diff_result.json を tokens.json に統合
  4. Figma へのインポート — figma-use CLI でノードを直接更新

変更種別ごとの自動化レベル:

変更種別反映方法自動化
色 (Color)tokens.json → Token Studio / Variables API✅ 自動
テキスト内容figma-use CLI でノード直接書き換え✅ 自動 (要 Figma Desktop 起動)
サイズ・座標figma-use CLI でノード直接書き換え✅ 自動 (要 Figma Desktop 起動)
タイポグラフィtokens.json → Token Studio✅ 自動
UI 構造追加・削除Plugin API (figma-use) で新規ノード生成⚠️ 複雑・スコープ外

実装予定スクリプト:

tools/
├── extract_flutter_diff.py   # git diff → diff_result.json
├── apply_diff_to_tokens.py   # diff_result.json → tokens.json（拡張版）
├── push_to_figma.sh          # figma-use でノード直接更新
└── sync_flutter_to_figma.sh  # 上記を一括実行

前提条件:

  • Windows 側で Figma Desktop が --remote-debugging-port=9222 付きで起動していること
  • figma-use (npm: dannote/figma-use) を WSL2 からインストール
  • 各ノードの ID を事前に Figma MCP で取得しておくこと

Figma ↔ Flutter 双方向連携の技術的評価

方向対象可能性手法
Figma → FlutterUI コンポーネント✅ 高Figma MCP + Claude Code
Figma → FlutterDesign Tokens✅ 高自前スクリプト tokens_to_dart.py（tokens.json → Dart）
Flutter → FigmaUI コンポーネント構造❌ ほぼ不可能ノード追加・削除は複雑すぎて自動化困難
Flutter → Figmaテキスト・色・サイズ⚠️ 条件付き自動figma-use (CDP) 経由で可能（Phase 6・保留中）
Flutter → FigmaDesign Tokens のみ⚠️ 手動操作tokens.json → Token Studio 手動 Import（現行・無料プラン制約）
完全な双方向同期全体❌ 不可能UI 構造変更は双方向対応不可

Flutter → Figma（逆方向）が困難な理由

  1. Figma REST API はノード修正に非対応 — 既存レイヤーのサイズ・位置・テキストを API で書き換えられない
  2. Widget ツリーは動的 — 条件分岐・ステート管理・レスポンシブ計算を静的な Figma ノードに復元できない
  3. 概念のミスマッチ — Flutter Widget (動的 UI 構築単位) と Figma Component (静的デザイン単位) は構造が根本的に異なる

補足: figma2flutter パッケージについて

figma2flutter は pub.dev で公開されている Design Tokens → Dart 変換パッケージ。 本 PoC では使用していない（自前スクリプトで同等機能を実装）が、将来の移行検討のために調査結果を記録する。

評価軸figma2flutter自前ツール (tokens_to_dart.py)
色✅ グラデーション・修飾子も対応✅ 対応
タイポグラフィ✅ 全属性対応⚠️ title スタイルのみ
スペーシング✅ EdgeInsets として生成⚠️ 数値定数のみ（未リンク）
ボーダー・シャドウ✅ 対応❌ 未対応
複数テーマ (Dark Mode等)✅ ITokens インターフェース❌ 非対応
双方向同期 (Dart→Figma)❌ 一方向のみ✅ dart_to_tokens.py で対応
カスタマイズ⚠️ Dart 知識が必要✅ Python で容易に拡張可
メンテナンス状況⚠️ v0.3.1-alpha（不安定）⚠️ PoC 段階

使い分けの目安:

  • PoC・プロトタイプ規模 → 自前ツールで十分。双方向同期の強みあり。
  • 本番・大規模デザインシステム・複数テーマ → figma2flutter を再評価。ただし alpha 版のため安定性に注意。
